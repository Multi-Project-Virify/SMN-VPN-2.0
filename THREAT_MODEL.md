# Threat Model

Честная граница того, от чего SMN VPN защищает, а от чего — нет. Проекты,
которые не публикуют это явно, вводят пользователей в заблуждение
относительно реального уровня защиты.

## Модель атакующего

### Защищаемся от:
- **Пассивный наблюдатель на одном хопе** (ISP, локальная Wi-Fi сеть,
  один скомпрометированный узел цепочки) — не видит содержимое трафика
  и не видит одновременно отправителя и получателя
- **Кража устройства в выключенном состоянии** — данные зашифрованы,
  ключи не извлекаются без сид-фразы/PIN
- **Угон Master-ID / подмена ключа в DHT** — невозможна без приватного
  Ed25519-ключа жертвы (см. `SignedKeyUpdate`)
- **Компрометация текущего сессионного ключа** — не раскрывает прошлый
  трафик (Perfect Forward Secrecy через эфемерный X25519 на сессию)
- **DPI по сигнатуре пакета** — частично (constant-size framing,
  traffic morphing), не гарантированно против продвинутого ML-анализа

### НЕ защищаемся (сознательно, на текущей версии):
- **Глобальный пассивный наблюдатель**, видящий трафик на входе И
  выходе цепочки одновременно (end-to-end timing correlation) —
  это открытая проблема для ЛЮБОЙ low-latency анонимной сети (Tor
  включая), не только нашей. Требует mixnet с существенной задержкой
  (как Nym) для полной защиты — вне scope текущей архитектуры,
  оптимизированной под low-latency VPN-use-case
- **Compromised endpoint** — если ваше устройство уже заражено
  malware/имеет root-компрометацию, VPN-слой это не решает
- **Кража устройства во включённом и разблокированном состоянии**
  во время активной сессии
- **Юридическое принуждение владельца exit-ноды** к логированию —
  RAM-only и no-logs architecture снижает возможность технически,
  но не отменяет юридического давления в конкретной юрisdiction
- **Cold-boot атака на физически изъятое включённое серверное железо**
  снижается (короткий TTL ключей), но не устранена полностью без
  аппаратной поддержки (см. README раздел Anti-forensics)

## Одобренные крипто-примитивы

| Назначение | Примитив | Крейт/либа |
|---|---|---|
| Key exchange (эфемерный) | X25519 | `x25519-dalek` |
| Подпись идентичности | Ed25519 | `ed25519-dalek` |
| AEAD-шифрование | XChaCha20-Poly1305 | `chacha20poly1305` |
| KDF | HKDF-SHA256 | `hkdf` + `sha2` |
| Зануление секретов | — | `zeroize` |

Самописные реализации любого из перечисленного — запрещены (см.
`CONTRIBUTING.md`).

## Обновление документа

Этот файл обновляется при каждом изменении крипто-архитектуры. Если
PR меняет модель угроз — обновление `THREAT_MODEL.md` обязательно
как часть того же PR, не отдельным follow-up.
