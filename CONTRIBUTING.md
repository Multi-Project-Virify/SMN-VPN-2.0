# Contributing to SMN VPN

## Перед первым PR

Отправляя Contribution в этот репозиторий, вы соглашаетесь, что она
лицензируется под Apache License 2.0 (см. `LICENSE`), без дополнительных
условий — это стандартная практика Apache-style проектов (аналог ICLA),
явный отдельный документ подписывать не нужно для мелких патчей.

Для **значительных вкладов** (новый модуль, доступ к приватным веткам,
crypto-review права) — см. раздел "Доступ выше Contributor" ниже.

## Как предложить изменение

1. Откройте Issue с описанием проблемы/фичи ДО того как писать код —
   особенно для чего-либо касающегося `smn-crypto-core` (крипто-ядро)
2. Форкните репозиторий, ветка `feature/<короткое-описание>`
3. PR с чётким описанием: что изменено и зачем
4. Изменения в `smn-crypto-core` требуют review минимум от одного
   существующего maintainer'а с криптографическим бэкграундом — обычные
   code-style ревью тут недостаточно

## Доступ выше Contributor (merge rights, релиз-ключи)

Мы **не** даём прямой push/merge доступ по одному PR, каким бы хорошим
он ни был. Модель ближе к Tor Project, не к "залогинься и пиши":

1. Нужен **Advocate** — существующий maintainer, который лично готов
   поручиться за вас на основе истории ваших PR
2. Обсуждение внутри команды maintainer'ов (минимум несколько дней)
3. Любой существующий maintainer может заблокировать кандидатуру
4. Только после этого — доступ к релизным GPG-ключам /
   merge-правам на защищённые ветки

Это медленнее, чем обычный open-source онбординг, но у нас нет
альтернативы для проекта, где скомпрометированный merge = скомпрометированная
приватность пользователей.

## Правила для крипто-кода конкретно

- Никаких самописных примитивов шифрования — только аудированные крейты/
  библиотеки (см. `THREAT_MODEL.md` за списком одобренных)
- Любой секрет (приватный ключ, сессионный ключ) обязан явно зануляться
  (`zeroize` в Rust) сразу после использования, не полагаясь на GC/drop
  без явного вызова
- PR с новым крипто-примитивом без ссылки на независимый аудит/спецификацию
  будет отклонён без обсуждения

## Code of Conduct

Будьте технически честны: если не уверены, что решение безопасно — прямо
напишите об этом в PR, не маскируйте неуверенность уверенным тоном.
