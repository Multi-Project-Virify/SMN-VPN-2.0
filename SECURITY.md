# Security Policy

## Reporting a Vulnerability

**Не открывайте публичный Issue для уязвимостей безопасности.** Публичный
репорт критического бага в handshake/шифровании даёт атакующим фору раньше,
чем выйдет патч.

Вместо этого:

1. Напишите на приватный контакт мейнтейнера (см. `MAINTAINERS.md`), в идеале
   зашифровав сообщение GPG-ключом мейнтейнера (fingerprint будет опубликован
   там же).
2. Опишите: какой компонент, воспроизводимые шаги, потенциальный impact.
3. Ожидайте подтверждение получения в течение 72 часов.

## Процесс

- Уязвимость подтверждается и оценивается по критичности (влияет ли на
  конфиденциальность трафика, деанонимизацию, RCE, DoS).
- Патч разрабатывается приватно в закрытой ветке.
- Публичный advisory и CVE (если применимо) публикуются одновременно с
  релизом патча — не раньше.
- Автор репорта указывается в благодарностях (если не просит анонимности).

## Что считается уязвимостью безопасности

- Любой баг, ломающий Perfect Forward Secrecy (утечка прошлых сессий при
  компрометации текущего ключа)
- Возможность подделать `SignedKeyUpdate` без приватного Ed25519-ключа
  (угон Master-ID)
- Утечка метаданных, деанонимизирующая пользователя (timing, packet size
  fingerprinting сверх задокументированного threat model)
- Любой RCE/memory corruption в Rust- или Kotlin-коде
- DoS, позволяющий вывести из строя значимую часть DHT

## Что НЕ считается уязвимостью безопасности (используйте обычные Issues)

- Проблемы производительности без security impact
- UI/UX баги
- Отсутствие фичи (feature request)

## Threat Model

См. `THREAT_MODEL.md` для явного описания: кого мы считаем атакующим, что
защищаем, и что **сознательно не защищаем** (например: активный
глобальный наблюдатель, компрометирующий все хопы цепочки одновременно —
вне модели угроз текущей версии).
